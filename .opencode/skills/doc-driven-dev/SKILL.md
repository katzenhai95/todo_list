# 文档驱动开发 (Doc-Driven Development)

## 核心原则

**文档是项目唯一的真相源。** 代码实现文档描述的功能，测试验证文档定义的验收标准。任何时候文档与代码不一致，以修正代码为优先——但最终必须让文档反映代码的实际行为。

---

## 开发流程（六阶段）

每当你接收到一个功能变更请求（新增、修改、删除），严格按以下六个阶段执行。每个阶段完成后再进入下一阶段。

### 阶段 0：现状摸底

**目标**：理解项目当前状态，不急于动手。

**操作**：
1. 检查 `docs/project-doc.md` 是否存在。不存在则跳至 [首次创建文档](#首次创建文档)。
2. 阅读文档全文，重点关注：
   - **功能库**：当前有哪些功能点，编号到多少
   - **架构**：模块依赖关系，关键类/方法的位置
   - **实现**：每个功能点对应的代码位置
   - **测试**：现有用例的编号规则、自动化还是人工
3. 浏览项目目录结构，确认文档中提到的文件是否存在，是否有未在文档中反映的新文件。
4. 以简洁的列表形式输出"当前状态摘要"给用户确认：
   ```
   当前功能点：F-001 ~ F-0xx（共 N 个）
   自动化用例：N 条
   人工用例：N 条
   待变更内容：[用户请求的变更摘要]
   ```

### 阶段 1：更新功能库

**目标**：在文档的功能库表中反映此次变更。

**操作**：

- **新增功能**：在功能库表末尾追加一行，编号为当前最大编号 +1（如 F-013）。
- **修改功能**：更新对应行的"功能描述"和"验收方式"列，不改变编号。
- **删除功能**：将该行"功能名称"改为 `~~原名称~~（已废弃）`，或直接删除该行（后续编号不变）。

**每行必须包含**：编号 | 功能名称 | 功能描述 | 验收方式。

**验收方式写法规范**：
- 用 → 连接因果链
- 描述用户可观察的行为，不是内部实现
- 人工验证的功能：最终验收方式以"用户自行检查..."开头（实际验收方式在测试用例节中详细展开）

完成后，向用户展示变更前后的功能库 diff，请求确认后再进入阶段 2。

### 阶段 2：更新测试用例

**目标**：为变更的功能编写或修改测试用例，确保每条功能都有对应的测试覆盖。

**自动化测试用例**（优先）：

- 位置：项目 `tests/` 目录下，按模块分文件（`test_models.py`、`test_storage.py` 等）。
- 新增用例时，在原测试文件末尾追加，或在必要时新建测试文件。
- 用例编号格式：`TC-F{功能编号}-{序号}`，如 `TC-F013-01`。
- 验收标准必须可自动判定（assert 语句）。
- 编写完用例后尝试运行，预期此时用例**失败**（因为功能代码尚未实现）——这证明用例有效。

**人工验证用例**（必要时）：

- 用于需要 GUI 交互、视觉反馈、系统级行为的功能。
- 用例编号格式：`TC-F{功能编号}-M`，多个用 `TC-F{功能编号}-M2` 等。
- 操作步骤编号化（1. 2. 3.）。
- 验收标准以**"用户自行检查{具体行为}，如果{具体标准}，则表明功能正常。"** 开头。

**更新文档**：

在 `docs/project-doc.md` 第四节中添加或修改对应的测试用例行。保持表格格式一致。

完成后向用户展示变更后的测试用例，确认后再进入阶段 3。

### 阶段 3：定位代码修改点

**目标**：根据功能库中的描述，精确找到需要修改的代码文件和位置。

**操作**：
1. 回顾阶段 1 中变更的功能点，提取关键行为描述。
2. 阅读文档第三节，找到对应功能点的实现位置（文件 + 方法）。
3. 阅读实际代码，确认文档记录的位置仍然准确。
4. 输出"代码修改计划"：

```
变更功能：F-0xx {功能名称}
涉及文件：
  - {文件}: {类/模块}，{新增/修改} {方法}
  - tests/{测试文件}: 新增 TC-F0xx-xx
修改策略：[简述修改思路]
```

获得用户认可后进入阶段 4。

### 阶段 4：实现代码

**目标**：最小化、精确地修改代码，不引入无关变更。

**约束**：
- 只修改与变更功能相关的代码。不顺手重构、不改格式、不动无关文件。
- 遵循项目现有代码风格和模式（参考同文件中已有的类似实现）。
- 每完成一个逻辑单元，运行一次相关的自动化测试。
- 新增代码需要添加简洁注释说明意图。

完成后输出变更摘要：修改了哪些文件、新增/修改了哪些方法。

### 阶段 5：验证与修复

**目标**：确保所有测试通过，不遗留回归问题。

**操作**：
1. 运行全量自动化测试：
   ```bash
   python -m pytest tests -v
   ```
2. 如有失败：
   - 分析失败原因（是新增用例的断言不正确，还是代码实现有 bug？）
   - 修复代码（不是修改测试用例，除非用例本身写错了）
   - 重新运行测试
   - 重复直到**全部通过**
3. 对人工测试用例，逐条提醒用户手动验证。

**失败处理阈值**：如果同一用例连续失败 3 次，暂停并向用户报告具体情况，征求指导。

### 阶段 6：收尾更新

**目标**：更新文档中除功能库和测试用例外其余部分，确保全文档反映最新状态。

**操作**：
1. 更新第三节（项目详细实现方式）：为变更的功能点新增或修改对应的小节。
2. 更新第二节（项目架构）：如果新增了模块/文件，更新依赖图和模块职责表。
3. 如果新增了依赖包，更新 `requirements.txt`。
4. 更新文档末尾的版本号和日期。
5. 输出最终变更摘要。

## 首次创建文档

如果项目中不存在 `docs/project-doc.md`：

1. 阅读项目全部代码，理解结构和功能。
2. 提取所有用户可感知的功能点，编号为 F-001 起。
3. 按 [模板](references/template.md) 生成完整文档。
4. 四个章节都必须填写，不能留空。
5. 使用模板中的表格格式，保持一致性。
6. 为每个功能点编写至少一个测试用例（自动化优先）。
7. 创建 `tests/` 目录并编写 pytest 用例。

## 文档一致性检查清单

每次开发完成后，逐项自查：

- [ ] 功能库表是否包含了所有功能？（代码中有但文档中无 = 遗漏）
- [ ] 功能库表是否包含了已废弃的功能？（文档中有但代码中无 = 过时）
- [ ] 每个功能点的验收方式是否可操作？（不是抽象描述）
- [ ] 每个功能点在第三节是否有对应的实现说明？
- [ ] 每个功能点在第四节是否有对应的测试用例？
- [ ] 自动化测试是否全部通过？（`pytest` 退出码为 0）
- [ ] 文档版本号和日期是否已更新？
